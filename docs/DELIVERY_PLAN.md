# 제안서·랜딩 페이지 전달 방식 (홈페이지 + PDF)

## 목표

| 산출물 | 전달 방식 |
|--------|------------|
| **홈페이지(랜딩)** | Git에 배포하여 URL로 제공 |
| **PDF(제안서)** | Figma 섹션별 페이지 + 추천 상품 페이지 → 회사명만 바꿔서 PDF로 전달 |

둘 다 **가능**하며, 아래처럼 연결하면 됩니다.

---

## 0. 전달 시 유의사항 (감안할 점)

### 0-1. 외부 링크에 대한 거부감

- 상대방이 **“밖의 링크를 눌러본다”**는 것에 거부감을 가질 수 있음 (보안·불편함·신뢰).
- **감안 방향**:
  - **우선**: **파일로 전달** (제안서 PDF, 랜딩 미리보기 PDF 등) → 이메일 첨부 등으로 “받은 파일만 열어보면 됨”.
  - **링크는 보조**: 링크를 쓸 때는 **자사 도메인** 또는 짧은 공식 URL로 제공해 “우리 회사가 제공하는 페이지”임을 분명히 하기.
  - **선택 제시**: “제안서는 PDF로 보내드리고, 랜딩은 (1) PDF 미리보기로 같이 드리거나 (2) 링크로 보시거나 선택하실 수 있습니다”처럼 **링크 없이도 내용을 볼 수 있는 경로**를 항상 하나 두기.

### 0-2. 제안서·랜딩 모두 깔끔하게 고퀄로

- **제안서**: Figma(또는 Canva)에서 **섹션별 레이아웃·타이포·여백**을 잡아두고, 회사명·추천 상품만 치환해 **고퀄 PDF**로 내보내면 충분히 깔끔하게 가능.
- **랜딩**:
  - **고퀄 웹으로 제공**하려면: **HTML 템플릿을 디자인 수준으로** 잡고(폰트, 색상, 섹션 구조, CTA 버튼 등), Git 배포 시에도 같은 퀄리티가 나오게 구성하면 됨. (가능함.)
  - **링크를 쓰기 어렵다면**: 랜딩 내용을 **“한 장/몇 장짜리 랜딩 미리보기 PDF”**로도 만들어서 제안서와 함께 첨부. “나중에 실제로 만들 페이지가 이렇게 보일 예정”이라고 보여주는 용도로 고퀄하게 가능.

정리하면, **외부 링크 거부감은 “파일(PDF) 우선 + 링크는 보조/선택”**으로 감안하고, **제안서와 랜딩 모두 Figma·좋은 HTML 템플릿·PDF 미리보기로 깔끔·고퀄**하게 낼 수 있음.

## 1. 홈페이지 → Git 배포

### 현재 상태
- Crew가 `landing_pages.md`를 생성 (회사별 헤드라인, 가치제안, 서비스, CTA 등 **마크다운**).
- 이걸 그대로 URL로 보여주려면 **HTML/정적 사이트**로 바꾼 뒤 **Git 저장소에 올려서** GitHub Pages / Netlify / Vercel 등으로 배포해야 함.

### 구현 방향 (택 1)

**A. 수동/반자동**
1. `landing_pages.md` 내용을 **HTML 템플릿**에 넣는 스크립트 작성 (회사명, 헤드라인, 본문 등 치환).
2. 생성된 HTML을 **별도 Git 저장소**에 넣고, GitHub Pages(또는 다른 서비스)로 배포.
3. 회사별 URL 예: `https://your-org.github.io/landings/동우문화` 등.

**B. Crew 확장**
1. **랜딩 페이지 출력을 HTML로** 내보내는 태스크/툴 추가 (마크다운 대신 또는 후처리로 HTML 생성).
2. 그 HTML을 지정한 폴더에 쓰고, 같은 폴더를 Git 저장소로 두고 푸시 → 배포 파이프라인(예: GitHub Actions)에서 자동 배포.

### 필요한 것
- HTML 템플릿 1개 (공통 레이아웃).
- `landing_pages.md` 또는 Crew 출력을 파싱해서 회사별 데이터 넣는 로직.
- 배포용 Git 저장소 + GitHub Pages(또는 Netlify/Vercel) 설정.

### 랜딩도 고퀄로 가능할까?
- **가능함.**  
  - **웹으로 제공할 때**: HTML 템플릿을 **디자인 수준**(타이포, 색상, 섹션 구조, CTA, 반응형)으로 한 번 잘 잡아두면, Git 배포해도 동일한 퀄리티로 노출 가능.  
  - **링크 없이 보여주고 싶을 때**: Figma 등에서 랜딩 화면을 **한 장/몇 장짜리 “랜딩 미리보기 PDF”**로 만들어 제안서와 함께 첨부. “실제로 만들 페이지가 이렇게 보일 예정”이라고 깔끔하게 전달 가능.

## 2. PDF → Figma 템플릿 + 이름만 바꿔서 전달

### 현재 계획 (요약)
- Figma에 **섹션별 페이지**를 이미 만들어 둠.
- **추천 상품 페이지** 추가.
- 회사마다 **회사명만 바꿔서** PDF로 내보내 전달.

### 가능 여부
- **가능합니다.**  
  Figma에서 텍스트 레이어만 회사명으로 수정한 뒤 **File → Export → PDF** 하면 됨.

### 운영 방식

| 방식 | 설명 |
|------|------|
| **수동** | Figma에서 회사명 텍스트 수정 → Export PDF → 클라이언트에게 전달. |
| **반자동** | Figma 플러그인 또는 스크립트로 “회사명” 플레이스홀더만 치환 후 Export (플러그인/API 사용). |
| **자동(고도화)** | Figma API로 노드 찾아서 텍스트만 바꾼 뒤, Export/이미지 가져와서 PDF 조합 (구현 난이도 높음). |

처음에는 **수동(이름만 바꿔서 PDF)** 으로 가고, 나중에 플러그인/API로 반자동화하는 걸 추천.

### Figma 측에서 해두면 좋은 것
- **회사명**이 들어갈 텍스트는 한 레이어로 두고, 이름만 바꾸기 쉽게 구성.
- 섹션별 프레임 = PDF 페이지 순서가 되도록 정리.
- “추천 상품” 페이지 추가 후, 회사별로 상품만 바꿀 수 있으면 더 좋음 (선택).

### 제안서 PDF를 깔끔·고퀄로 내려면
- Figma(또는 Canva)에서 **섹션별로 레이아웃·타이포·여백·색상**을 한 번 정리해 두고, 회사명·추천 상품 등만 치환해 Export PDF 하면 **고퀄 제안서**로 충분히 가능.
- 한 번 템플릿을 잘 잡아두면, 이후에는 내용만 바꿔서 동일한 퀄리티로 계속 쓸 수 있음.

### 2-2. 자동화 고퀄 PDF: HTML/CSS → Playwright (추천)
- **ReportLab** 한계(시각적 퀄리티, 2단 레이아웃, 차트)를 넘기려면 **HTML/CSS로 레이아웃을 만들고 Playwright로 PDF 인쇄**하는 방식을 쓸 수 있음.
- 프로젝트에 **`generate_pdf_playwright.py`** 가 포함되어 있음.  
  - **입력**: `proposal.md`, `landing_pages.md` (기존과 동일).  
  - **동작**: 마크다운 파싱은 `generate_pdf.py`와 동일하게 재사용하고, **HTML 템플릿 + CSS**로 렌더링한 뒤 **Chromium으로 PDF** 출력.  
  - **실행**:  
    `pip install playwright` 후 `playwright install chromium`,  
    `python generate_pdf_playwright.py [--proposal proposal.md] [--landing landing_pages.md] [--company "회사명"] [--out output]`  
  - 출력 파일은 `output/` 아래 `{회사명}_제안서_{날짜}_playwright.pdf` 형태로 저장됨.
- CSS로 커버·2단·카드·테이블을 자유롭게 꾸밀 수 있어 **Canva/디자인 툴에 가까운 퀄리티**로 자동 생성 가능.

## 3. Crew 출력과의 연결

| Crew 출력 | 홈페이지(Git) | PDF(Figma) |
|-----------|----------------|------------|
| `landing_pages.md` | ✅ HTML 생성 시 회사별 카피/헤드라인/CTA 소스로 사용 | — |
| `proposal.md` | 선택: 제안 요약을 랜딩 하단 등에 넣을 수 있음 | ✅ Figma 제안서 섹션 내용 참고용 |
| `marketing_plan.md` | — | ✅ “추천 상품” 페이지 채우는 데 사용 |

- **홈페이지**: `landing_pages.md` → HTML 템플릿 치환 → Git 배포.
- **PDF**: Figma 템플릿(섹션 + 추천 상품) + 회사명만 치환 → PDF Export → 전달.

## 4. 다음 단계 제안

1. **홈페이지**
   - 랜딩 1개 회사 기준으로 **HTML 템플릿 1개** 만들기.
   - `landing_pages.md` 파싱해서 회사별 HTML 생성하는 **스크립트** 작성.
   - 배포용 **Git 저장소 + GitHub Pages** 설정 후, 한 회사만 먼저 배포해 보기.

2. **PDF**
   - Figma에 **추천 상품 페이지** 추가.
   - 한 회사로 **이름만 바꿔서 PDF** 한 번 만들어 보기.
   - 필요하면 나중에 Figma 플러그인/API로 “회사명 치환 → Export” 반자동화.

이 순서로 하면 “홈페이지는 Git으로 URL 제공, PDF는 Figma에서 이름만 바꿔서 전달” 구조를 그대로 쓸 수 있습니다.

## 5. 이외에 추천하는 방법

아래는 지금 방식에 **더하거나 대체**해서 쓸 수 있는 옵션입니다.

### 5-1. 랜딩/제안서 공유용

| 방법 | 장점 | 단점 |
|------|------|------|
| **Notion 공개 페이지** | 이미 Notion 쓰고 있음. 제안 요약·랜딩 초안을 Notion 페이지에 넣고 **Share → Publish to web** 하면 URL 하나로 공유. 수정하면 바로 반영. | 디자인 자유도 낮음, 브랜드 감성은 제한적. |
| **Notion DB + 페이지 본문** | CRM 로그할 때 같은 DB에, 제안서 요약을 페이지 본문이나 속성으로 넣어두고 링크 공유. 팀 내에서 한 곳에서 관리. | 외부 클라이언트에게는 "Notion 링크"로 보내게 됨. |
| **Google Docs 템플릿** | 제안서를 Google Doc 템플릿으로 만들고, 회사명/내용만 치환한 복사본 생성 후 "보기 전용 링크"로 공유. 협업·댓글 편함. | PDF만큼 "완성본" 느낌은 약함. |

### 5-2. PDF 만들기 (Figma 대안/보조)

| 방법 | 장점 | 단점 |
|------|------|------|
| **Reportlab으로 PDF 자동 생성** | 프로젝트에 이미 `reportlab` 있음. `proposal.md` / `landing_pages.md` 파싱해서 **코드로 PDF 생성** 가능. 회사명·섹션만 바꿔도 되니 자동화·일관성 좋음. | Figma보다 디자인 표현은 단순함. |
| **Canva** | Figma처럼 템플릿 + 이름만 바꿔서 PDF. 비디자이너도 쓰기 쉬움, 템플릿 많음. | Figma만큼 픽셀 단위 제어는 아님. |

### 5-3. 전달·공유 채널

| 방법 | 장점 | 단점 |
|------|------|------|
| **아웃리치 이메일 + 첨부/링크** | 이미 `outreach_emails.md` 있음. 제안서 PDF 또는 랜딩 URL을 **이메일 본문/첨부**로 넣어서 보내기. 클라이언트가 한 번에 받음. | 이메일만 의존. |
| **Slack** | 팀 내에서 "이번 주 제안서/랜딩 목록"을 슬랙으로 요약 전송. 봇으로 `proposal.md` 요약이나 Notion 링크 전달 가능. | 외부 클라이언트 전달용은 아님. |

### 5-4. 추천 조합 (외부 링크 거부감 감안)

- **기본**: **제안서 PDF(Figma) + 랜딩 미리보기 PDF**를 이메일 첨부로 전달 → 링크 없이 파일만으로 깔끔·고퀄 전달. 필요 시에만 “실제 랜딩은 이 URL에서 보실 수 있습니다”(자사 도메인 권장) 보조로 안내.
- **지금 계획 + 링크 보조**:  
  홈페이지(Git) URL은 **선택 옵션**으로 두고, 기본은 PDF 첨부. Notion 공개 페이지 등 외부 링크는 “추가로 보시려면” 수준으로만 제시.
- **PDF를 더 자동화하고 싶다면**:  
  Figma는 "최종 고퀄 버전"용으로 두고, **Reportlab**으로 회사별 제안서 PDF를 자동 생성해 두었다가, 필요 시 Figma PDF로 교체하거나 둘 다 제공.
- **디자인 손이 적게 가면**:  
  **Canva** 템플릿으로 제안서·랜딩 미리보기 모두 만들고 회사명만 바꿔서 PDF 내보내기. Figma보다 진입 장벽 낮음.

### 5-5. 최종 산출물: 이메일(PDF 첨부) + 노션(PDF 링크) 한 번에

- **목표**: 고객에게는 **이메일로 PDF 첨부**해서 보내고, **우리 쪽 Notion**에는 같은 제안서 PDF 링크를 회사별 페이지에 넣어 두기.
- **방법**  
  1. `generate_pdf_playwright.py`로 제안서 PDF 생성 → `output/`에 저장.  
  2. `send_emails.py --contacts contacts.csv --send --sync-notion` 실행.  
     - **이메일**: 수신자에게 해당 회사 PDF를 **첨부**해서 발송.  
     - **노션**: `NOTION_PDF_BASE_URL`이 설정되어 있으면, 발송 후 자동으로 `sync_pdf_to_notion.py`를 실행해 각 회사 Notion 페이지의 **제안서 PDF(URL)** 속성에 링크를 채움.  
  3. **NOTION_PDF_BASE_URL**: Notion은 파일을 직접 올리지 않고 **URL만** 저장함. `output/` 폴더를 OneDrive·Google Drive·SharePoint 등에 올리고 **폴더 공유 링크**를 만든 뒤, 그 기준 URL을 `.env`에 `NOTION_PDF_BASE_URL=https://...` 로 넣으면, 스크립트가 `기준URL/파일명.pdf` 형태로 링크를 채움.
- **정리**: 최종 산출물 = **이메일(PDF 첨부)** + **노션(제안서 PDF 링크)**. 한 번 발송하면 둘 다 반영되도록 되어 있음.

## 6. 냉철한 수요 점검 + 과감한 추천

### 6-1. 실제 수요가 있는지

- **진짜 수요**는 “제안서 포맷이 여러 가지”가 아니라 **회신률·미팅 예약**을 올리는 것에 가깝다.
- 상대(인쇄소·중소 사업자 등)는 보통:
  - **빨리 훑어보고** → **가격/다음 단계가 명확한지** → **신뢰(사례·근거)** 를 본 뒤 결정한다.
- 따라서 “전달 방식”보다 **내용·구성·한 장 요약·가격·사례**가 수요에 더 직결된다.

### 6-2. 산출물별 수요 (냉정 평가)

| 산출물 | 수요 강도 | 비고 |
|--------|-----------|------|
| **제안서 PDF (고퀄)** | 높음 | 깔끔한 PDF는 기본 기대. 유지·강화 권장. |
| **1페이지 요약(첫 장)** | 높음 | 긴 제안서는 안 읽힘. “한 장에 핵심만” 수요 큼. |
| **가격·다음 단계 명시** | 높음 | 흐릿하면 회신률 떨어짐. 4주 예산·월 비용·연락 CTA를 분명히. |
| **짧은 사례/근거 1개** | 높음 | “비슷한 업종 적용” 한 줄만 있어도 신뢰 상승. |
| **랜딩 미리보기 PDF** | 중간 | “우리가 뭘 만들어줄지” 시각적으로 보여주기 좋음. 필수는 아니고 선택 강화용. |
| **랜딩 실제 URL (Git 배포)** | 낮음~중간 | “링크 눌러서 보기” 수요는 상대에 따라 다름. 상당수는 PDF만으로도 충분. **선택 옵션**으로 두는 게 맞음. |

### 6-3. 과감하게 추가할 것

1. **1페이지 실행 요약 (첫 장)**  
   제안서 **맨 앞에 한 장**: 회사명, 현재 상태 한 줄, 제안 한 줄, 추천 채널 1~2개, 4주 예산/월 비용, 다음 단계(전화/미팅 요청). 나머지 페이지는 상세용.  
   → “한 장만 읽어도 결론이 보인다”가 수요에 맞음.

2. **가격·다음 단계를 반드시 명시**  
   proposal/태스크에서 **4주 테스트 예산**, **월 예상 비용**, **“지금 전화/캘릭 예약”** 같은 CTA를 빼먹지 않고 Figma 제안서에 넣기.  
   → 회신·미팅 전환에 직접 기여.

3. **사례/근거 1개**  
   “인쇄·유사 업종 적용 사례” 또는 “이런 채널에서 이런 결과”를 **한 블록**으로 Figma 제안서에 넣기. 텍스트 2~3줄이어도 됨.  
   → 신뢰·수요 모두 올라감.

4. **제안서 + 랜딩 미리보기 한 PDF**  
   가능하면 **제안서와 랜딩 미리보기(1~2장)** 를 **한 개 PDF**로 합쳐서 “첨부 1개만 열면 끝”으로 주기.  
   → 열어볼 동기·완독률이 올라감.

### 6-4. 과감하게 줄이거나 옵션으로 둘 것

1. **랜딩 URL(Git 배포)**  
   냉정히 보면 “실제 URL까지 보여준다”는 수요는 상대에 따라 제한적.  
   → **옵션**으로 두고, 기본은 **제안서 PDF(+ 랜딩 미리보기 포함)**. “원하시면 실제 페이지 URL도 드립니다” 수준.

2. **산출물 개수**  
   제안서 PDF / 랜딩 미리보기 PDF / 랜딩 URL을 **각각 따로** 주기보다,  
   → **한 PDF(요약 + 상세 + 랜딩 미리보기)** + (필요 시) 링크 하나.  
   → 전달이 단순해지고, “뭘 먼저 봐야 하지” 하는 소모가 줄어듦.

3. **Crew/문서 쪽**  
   `proposal_task` 기대 출력에 **“첫 장 1페이지 요약”**, **“가격·next step 명시”**, **“사례 1개”** 를 명시해 두면 Figma에 옮길 때 빠짐없이 반영하기 쉬움.  
   → **반영 완료**: `config/tasks.yaml`의 `proposal_task`에 위 항목이 포함되어 있음.

### 6-5. 정리 (과감한 한 줄)

- **수요는 “회신·미팅”에 있으니, “한 장 요약 + 가격·CTA·사례 한 줄”을 넣은 고퀄 제안서 PDF 하나에 올인하고,**  
  **랜딩은 그 PDF 안 1~2장 미리보기 + (선택) URL** 로 가져가는 구도가 가장 수요에 맞고 운영도 가볍다.

## 7. 해외 수요·해외 진출

### 7-1. 이런 수요가 있는 해외가 있나?

- **있음.**  
  “리드 발굴 → 웹사이트 검토 → 랜딩/마케팅 제안 → 제안서 → 아웃리치 → CRM 로깅” 파이프라인은 **업종·지역을 바꾸기만 하면** 해외에서도 동일하게 쓰인다.
- **수요가 큰 곳**:
  - **미국·캐나다·영국·호주**: SMB(소상공인) 수가 많고, “로컬 비즈니스 마케팅/웹사이트/광고 대행”을 파는 에이전시·프리랜서가 많음. 리드 리서치·제안서 자동화 수요가 지속적으로 있음.
  - **업종**: 인쇄만이 아니라 **식당, 클리닉, 법률/회계 사무소, 이사·인테리어, 피트니스, 살롱** 등 “지역 기반 SMB” 전반. 각 업종별로 `lead_query`, 채널(Google Ads, Meta, Yelp, local SEO 등)만 바꿔주면 됨.
  - **B2B 에이전시**: “우리 에이전시가 이걸 쓰면 클라이언트 제안을 10배 빠르게 만든다”는 화이트라벨/내부 도구 수요도 있음.

### 7-2. 아예 해외로 뛰어버릴까?

- **옵션 정리**:
  1. **한국 유지 + 영어/해외 한정 실험**  
     Crew 프롬프트·산출물을 **영어**로 바꾸고, `lead_query`를 해외 지역·업종으로 두고 테스트 (예: "printing companies Chicago", "dental clinics UK"). 국내와 동일 구조, 언어·타깃만 해외.
  2. **해외 1개 시장 집중**  
     예: 미국 SMB 마케팅 에이전시용. 채널을 Google Ads, Meta, Yelp, local SEO 등으로 고정하고, 제안서·아웃리치 포맷을 미국 습관에 맞춤. 수요 검증 후 확대.
  3. **에이전시 화이트라벨**  
     “Sales Factory를 그대로 쓰는 게 아니라, 에이전시가 자신만의 브랜드로 고객에게 제안서·리드를 쏘는 도구”로 판매. 해외 에이전시가 더 단가·규모가 클 수 있어서 수요 있음.
  4. **완전 해외 전환**  
     국내 인쇄 영업을 줄이고, 영어 + 해외 업종/지역으로만 운영. 리스크는 고객 지원·결제·법인 등이지만, 단가·시장 크기는 해외가 더 클 수 있음.

- **추천**:  
  **먼저 1번(한국 유지 + 영어/해외 실험)** 으로 Crew 출력만 영어로 돌려보고, 제안서·아웃리치 품질을 본 뒤, 반응이 좋으면 2번(해외 1개 시장) 또는 3번(에이전시 화이트라벨)으로 확장하는 게 무난함. “아예 해외로만”은 검증 후 결정해도 늦지 않음.
