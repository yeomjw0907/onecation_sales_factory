# AI 오케스트레이션 — 명확한 지시 가이드

CrewAI 기반 파이프라인에서 **명확한 지시**를 하려면 어떻게 해야 하는지, 현재 방식이 맞는지 정리합니다.

---

## 1. 오케스트레이션에서 “명확한 지시”란

| 원칙 | 설명 |
|------|------|
| **입력·출력 계약** | 각 태스크가 **무엇을 받고**, **무엇을 내는지** 형식·개수까지 명시. |
| **단일 기준(Source of Truth)** | “몇 개”를 만들지 같은 숫자는 **한 곳에서만 정해지고**, 이후 단계는 그걸 따름. |
| **산출물 형식 고정** | 다음 단계(또는 스크립트)가 파싱할 수 있도록 **첫 줄·구분자·키 이름** 등을 고정. |
| **검증 가능** | 산출물을 스크립트로 열어 **N과 실제 개수**를 비교해 검증할 수 있음. |

---

## 2. 현재 방식 정리

### 2-1. 하고 있는 것

- **“대상 회사 수: N”** 을 각 단계 산출물 **맨 앞**에 적도록 지시.
- **N에 맞춰 생성**: “정확히 N개만 작성, N개 초과·미만 금지.”
- **형식 고정**: proposal/landing은 회사 구분을 **`# 회사명` (H1)** 한 줄로만 하도록 해서, PDF 스크립트가 파싱 가능하게 함.

이것은 오케스트레이션 관점에서 **올바른 방향**입니다.

- **개수 명시** → 다음 단계가 “몇 개 만들어야 하는지” 알 수 있음.
- **형식 고정** → 후처리(PDF, Notion 동기화)가 같은 형식을 기대하고 동작함.
- **검증** → PDF 스크립트에서 “대상 회사 수: N”과 실제 H1 블록 수를 비교해 안내 메시지를 낼 수 있음.

### 2-2. 단일 기준(Single Source of Truth)

**N의 기준**은 한 곳으로 모는 것이 좋습니다.

- **권장**: **website_audit_task** 산출물의 **첫 줄 “대상 회사 수: N”** 을 **정식 N**으로 둠.
- **이유**: 리드 수집(lead) 후 감사(audit)에서 실제로 제안·PDF·Notion에 쓸 “대상 회사”가 확정되므로, 이 단계의 N이 이후 전체 파이프라인의 개수를 결정하는 것이 자연스러움.

따라서 **proposal, landing, email, notion** 단계에서는 모두  
“**N은 website_audit_task 산출물 맨 앞의 ‘대상 회사 수: N’과 동일한 숫자**”라고 지시하는 것이 좋습니다.

---

## 3. 지시 문구 패턴 (명확하게 쓰는 법)

### 3-1. 목적 먼저

- **나쁜 예**: “테이블을 작성하라.”
- **좋은 예**: “이 테이블의 행 수(N)가 제안서·PDF·이메일·Notion의 **생성 개수**가 됨. 따라서 …”

→ **왜** 그 형식/개수가 중요한지 한 줄로 적어 주면, 에이전트가 지키기 쉬움.

### 3-2. 산출물 형식 명시

- **저장 위치**: “산출물은 OO.md / 태스크 output_file로 저장됨.”
- **첫 줄**: “첫 줄에 반드시 ‘대상 회사 수: N’ (N=숫자) 명시.”
- **구조**: “이어서 N개의 블록. 각 블록은 `# 회사명` (H1) 한 줄로 시작.”

→ **파싱 가능한 계약**이 됨.

### 3-3. 개수 제약

- “**정확히 N개**만 작성. N개 초과·미만 금지.”
- “N은 **website_audit_task 산출물 첫 줄의 숫자**와 동일해야 함.”

→ **단일 기준**과 **검증 가능성**이 생김.

### 3-4. 다음 단계 입장

- “이 파일은 **generate_pdf_playwright.py**의 직접 입력이며, **회사당 1개 PDF**가 생성됨.”

→ **다운스트림 동작**과 맞춰서 지시할 수 있음.

---

## 4. 현재 방식이 “맞는지” 체크

| 항목 | 상태 | 비고 |
|------|------|------|
| 개수(N) 명시 | ✅ | “대상 회사 수: N”으로 명시하도록 지시함. |
| N에 맞춰 생성 | ✅ | “정확히 N개, 초과·미만 금지”로 지시함. |
| 형식 고정(H1) | ✅ | PDF/동기화 스크립트가 파싱할 수 있는 형식으로 고정함. |
| 단일 기준 | ⚠️→✅ | **N의 기준을 website_audit_task로 통일**하면 더 명확함. |
| 검증 가능 | ✅ | PDF 스크립트에서 N vs 실제 블록 수 비교·안내 가능. |

**결론**:  
- “각 AI 산출물을 테이블/파일로 저장하고, 그 **갯수(N)를 명시**한 뒤, **N에 맞춰 생성**하라”는 **방법은 AI 오케스트레이션 관점에서 맞는 방법**입니다.  
- 여기에 **N의 단일 기준(website_audit_task)** 을 태스크 지시에 넣으면, 더 명확한 오케스트레이션에 가깝습니다.

---

## 5. 추가로 할 수 있는 것 (선택)

- **검증 스크립트**: Crew 종료 후 `proposal.md`·`landing_pages.md`를 읽어, “대상 회사 수: N”과 실제 `#` (H1) 개수를 비교하고, 불일치 시 경고 또는 실패로 처리.
- **입력/출력 스키마 문서화**: 각 태스크별로 “입력: OO 태스크의 N, 테이블” / “출력: 첫 줄 N, 이어서 …” 를 한 줄씩 정리해 두면, 팀·유지보수에 유리함.

이 문서는 `config/tasks.yaml` 의 지시와 함께, **오케스트레이션 설계의 기준**으로 사용하면 됩니다.
